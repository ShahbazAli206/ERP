# ERP Demo — Frontend (`apps/web`)

Next.js 16 (App Router, Turbopack) + React 19 + TypeScript + Tailwind v4 + shadcn/ui.
This is the frontend for the Import Business Management System demo — see the root
`IMPLEMENTATION_PLAN.md` for the overall project plan and `apps/api` for the backend.

**Read `AGENTS.md` first.** This Next.js version has real breaking changes vs. typical
training data (see "Next 16 gotchas" below) — check `node_modules/next/dist/docs/`
before assuming a Next 14/15 pattern still applies.

## Getting started

```bash
# from the repo root
npm install
cp apps/web/.env.example apps/web/.env.local   # already done; edit if your API runs elsewhere
npm run dev --workspace=apps/api    # backend, port 4000
npm run dev --workspace=apps/web    # frontend, port 3000
```

Demo logins (all roles, password `Demo@1234`): `admin@erp.local` (Super Admin, all
permissions), `procurement@erp.local`, `inventory@erp.local`, `sales@erp.local`,
`accounts@erp.local`, `executive@erp.local` (view-only everywhere).

### Adding a Phase 8 module page — the pattern to copy

1. **`src/features/<module>/api.ts`** — typed functions calling `apiClient`
   (`src/lib/api-client.ts` has a full worked example in its top-of-file docstring).
2. **`src/features/<module>/schemas.ts`** — Zod schemas for any create/edit forms.
3. **React Query hooks** (colocated in `api.ts` or a sibling `hooks.ts`) wrapping the
   `api.ts` functions — `useQuery`/`useMutation`, following `src/features/auth/use-auth.tsx`'s
   pattern for anything that needs to be read from multiple components.
4. **`src/app/(app)/<module>/page.tsx`** — replace the `<ComingSoon />` placeholder with:
   - `<PageHeader title="..." description="..." actions={...} />` at the top.
   - `<DataTable />` (`src/components/shared/data-table.tsx`) for list views — it owns
     rendering only; your page's hook owns the actual paginated fetch.
   - `<StatCard />` / `<ChartCard />` (`src/components/shared/`) for dashboard-style
     summaries — `ChartCard` has a full usage example including the validated chart
     color palette in its docstring.
   - Forms: `useForm` + `zodResolver` + the wrappers in
     `src/components/shared/form-fields.tsx` (`TextFormField`, `SelectFormField`,
     `TextareaFormField`, `CheckboxFormField`) — see `src/features/auth/login-form.tsx`
     for the complete reference example.
   - File attachments: `<FileUpload />` (`src/components/shared/file-upload.tsx`).
5. Nothing to change in `nav-config.ts`, `proxy.ts`, or `(app)/layout.tsx` unless the
   route itself changes — the shell, nav visibility, and breadcrumbs are automatic.

### Auth / session model

- The API returns the JWT in the login response body (no `Set-Cookie`); the frontend
  sets its own **plain (non-httpOnly) cookie** (`erp_token`, see `src/lib/session.ts`)
  after login. This is a deliberate demo tradeoff — a plain cookie is readable by both
  client JS (to attach `Authorization: Bearer <token>`) and `src/proxy.ts` (server-side
  redirect gate), at the cost of XSS exposure an httpOnly-cookie-plus-route-handler-proxy
  setup would avoid. Acceptable here; call it out if hardening this for real use.
- `src/proxy.ts` (Next 16's renamed `middleware.ts`) does an **optimistic** check —
  cookie presence only, not signature/expiry — redirecting to `/login` (or away from
  `/login` if already authenticated) before any page renders, so there's no flash of
  protected content.
- Actual token validity is enforced by the API on every request; a 401 anywhere is
  caught globally by `src/lib/query-provider.tsx`'s `QueryCache.onError`, which clears
  the session and redirects to `/login`. Between the proxy gate and this global
  handler, both "never logged in" and "session expired mid-use" are covered.
- `src/features/auth/use-auth.tsx`'s `AuthProvider` (mounted in `(app)/layout.tsx`,
  not the root layout) fetches `GET /api/auth/me` and is the single source of the
  current user/permissions — `useAuth()` anywhere under the protected shell.

## Next.js 16 gotchas (read before writing App Router code)

- **`middleware.ts` → `proxy.ts`.** Renamed, not just aliased — the exported function
  is `proxy`, not `middleware`. `middleware.ts` still works but is deprecated. Proxy
  is Node.js-runtime-only (no Edge option); see
  `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/proxy.md`.
- **`cookies()`/`headers()` are fully async** (`await cookies()`) — synchronous access
  (allowed with a deprecation warning in 15) is fully removed in 16.
- **`params`/`searchParams` are `Promise`s** in Server Component page/layout props —
  `{ params: Promise<{ id: string }> }`, `await params`. Client Components receiving
  them use React's `use()` hook instead of `await`.
- **`useSearchParams()` still needs a `<Suspense>` boundary** around the Client
  Component using it, or the production build fails — see `src/app/login/page.tsx`
  for a real example (the login form reads `?from=` for the post-login redirect).
- **`next/image`**: `images.domains` is deprecated (`remotePatterns` instead),
  `minimumCacheTTL` default jumped from 60s to 4h, default `qualities` is now `[75]`
  only. Not used anywhere in this app yet, but relevant if a module needs remote
  images later (e.g. product photos via S3 in Phase 4's storage abstraction).
- **shadcn's registry no longer ships the classic `Form`/`FormField` component** (the
  one tightly coupled to react-hook-form) for this style/base combination — `npx
  shadcn add form` silently no-ops. The replacement is a layout-only `field` primitive
  (`Field`/`FieldLabel`/`FieldDescription`/`FieldError`, no RHF binding). This repo's
  RHF binding layer on top of it is `src/components/shared/form-fields.tsx` — use that,
  don't reach for the old `Form`/`FormField` API from memory.
- **This shadcn "base" library is Base UI (`@base-ui/react`), not Radix.** Polymorphism
  uses a `render` prop (`<SidebarMenuButton render={<Link href="..." />}>`), not
  Radix's `asChild`. Exception: the actual `Button` primitive enforces button
  semantics and should **not** be repurposed for links via `render` — style an `<a>`/
  `<Link>` directly with the exported `buttonVariants()` instead (see
  `src/components/layout/notification-bell.tsx`).
- **`eslint-config-next` 16 ships a React Compiler ESLint rule
  (`react-hooks/set-state-in-effect`)** that flags the classic
  `useState(false)` + `useEffect(() => setMounted(true), [])` hydration-guard pattern
  as an error — including in shadcn's own generated `use-mobile.ts`. Use
  `useSyncExternalStore` instead (see `src/hooks/use-mounted.ts` and the rewritten
  `src/hooks/use-mobile.ts`), not an eslint-disable.
- **React Query + TanStack Table**: `useReactTable()` triggers a `react-hooks/incompatible-library`
  *warning* (not an error) from the React Compiler — expected, TanStack Table's API
  isn't compiler-memoizable yet; safe to ignore.
- **Two real bugs found in shadcn's own generated code** (not introduced by this repo,
  but fixed here since a naive copy-paste would reproduce them):
  1. `components/ui/sidebar.tsx`'s `SidebarMenuSkeleton` picked its loading-bar width via
     `useState(() => Math.random()...)`, which differs between the SSR render and the
     client's hydration render and throws a hydration-mismatch warning on every page
     load while the sidebar is loading. Fixed by making `width` a prop with a fixed
     default instead of internal randomness (see `app-sidebar.tsx`'s `SKELETON_WIDTHS`
     for varying-but-deterministic values).
  2. Base UI's `Menu.GroupLabel` (which shadcn's `DropdownMenuLabel` wraps) throws
     `"MenuGroupContext is missing"` at runtime if rendered outside a `<Menu.Group>` —
     i.e. `<DropdownMenuLabel>` must always be wrapped in `<DropdownMenuGroup>`, unlike
     the old Radix-based dropdown where a bare `DropdownMenuLabel` worked fine. See
     `components/layout/user-menu.tsx` for the corrected pattern. This only surfaces at
     runtime (not typecheck/lint/build), so it's easy to miss without live verification.
  3. `<Select>` (`components/ui/select.tsx`, wrapping Base UI's `Select.Root`) only shows
     the correct label in its **closed** trigger when given an `items={[{value, label}]}`
     prop — without it, `<SelectValue>` falls back to the raw stored value once the popup
     (and its `<SelectItem>` children) unmounts, e.g. showing a raw id instead of a name,
     or `"ALL"` instead of "All statuses". **`SelectFormField` in
     `components/shared/form-fields.tsx` now passes `items={options}` automatically** —
     if you use the RHF-bound `SelectFormField`, you get this for free. If you build a
     **raw** (non-form) `<Select>` directly — e.g. a list-page filter dropdown — you must
     pass `items` yourself: `<Select items={options} ...>`. This only surfaces once the
     popup closes (typecheck/lint/build don't catch it), so always close a select after
     picking a non-default option during verification and check the trigger's displayed
     text, not just that the popup itself worked.

## Charts & color

`src/components/shared/chart-card.tsx` wraps shadcn's Recharts integration. The base
theme (`base-nova`/neutral) is intentionally grayscale for UI chrome, but chart series
use a separate, validated categorical palette (`--chart-1` … `--chart-8` in
`globals.css`) — colorblind-safe adjacent-pair separation and WCAG-checked per mode,
picked via the project's data-viz color methodology rather than shadcn's generated
grayscale chart defaults. Always pair color with a legend/direct labels; never rely on
hue alone (see the docstring in `chart-card.tsx` for specifics). `src/lib/status-colors.ts`
is a separate fixed good/warning/serious/critical palette for "needs attention" states
— never reuse chart series colors for status, and vice versa.
